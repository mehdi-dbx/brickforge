1. Vision & Core Shift

Vision

BrickForge goes from a locally-cloned project into a cloud-native SaaS product. The user never clones a repo, never runs a terminal, never installs anything. They open a browser, create an agent, deploy it. Done.

The user also needs access to the generated code. They must be able to CI/CD it, edit it in an IDE, and own it. The Setup App is a PROJECT GENERATOR, not just a deployer.

The Core Shift

Before

User clones repo → has all code locally → runs scripts → deploys from local machine

After

User opens browser → Setup App does everything → code never touches user's machine (unless they want it to)

Setup App = The Product

The visual Setup App (currently visual/) IS the product. It:

  • Creates and manages .forge projects (multi-project: save/load/switch)
  • Walks users through domain setup via LLM-assisted wizard
  • Provisions UC data (tables, functions, procedures)
  • Deploys the Agent App to Databricks
  • Generates a git-ready project the user can own
  • Runs as a Databricks App itself (or locally for dev)

B: User's DBX App User deploys Setup App to their own workspace Production use DBX App SP (user's permissions)

3. .forge Schema

What It Is

  • Internal serialization format -- machine-generated, machine-consumed
  • One .forge file per project
  • User NEVER writes YAML by hand -- the Setup App wizard generates it
  • Contains BOTH domain content (tables, SQL, prompts) AND infrastructure config (schema, warehouse, genie IDs)

Domain Content

  • Domain description
  • Table definitions (DDL, seed data references)
  • UC function SQL (inline or sidecar files)
  • Stored procedure SQL
  • Tool definitions (declarative: 7 patterns)
  • System prompt + knowledge base + starter prompts
  • KA configurations
  • Eval dataset + scorer config
  • UI dashboard config

Where It's Stored

ModePathFormat
SaaS (A/B) /Volumes/{catalog}/{schema}/brickforge/stash/airops.forge.zip Single zip archive
Local (C) stash/airops/ on filesystem Directory (unzipped for dev)

What's IN the archive

.forge YAML, SQL files, prompts, tool specs, KA configs, app.yaml, databricks.yml -- all small text files. Typically under 1MB compressed.

What's NOT in the archive

PDFs (stay in UC Volume, referenced by path). Large datasets (already in UC tables as Delta).

How it works

  1. Download one zip from UC Volume (single API call: w.files.download())
  2. Open in memory (zipfile.ZipFile(io.BytesIO(bytes)) in Python, AdmZip(buffer) in Node)
  3. Read any file on demand -- no disk extraction, no temp files
  4. When saving: update entries in memory, upload zip back (single API call: w.files.upload())

Implementation: Python zipfile (stdlib, zero deps). Node.js adm-zip (50KB, zero transitive deps). Rock solid.

FORGE_STASH_DIR: Stash-Aware Data Provisioning (Implemented)

When the env var FORGE_STASH_DIR is set, the provisioning scripts resolve all data file paths relative to that directory instead of the default local paths. Implemented in:

  • data/create_all_assets.py -- tables DDL + seed CSV
  • data/create_all_functions.py -- UC function SQL files
  • data/create_all_procedures.py -- stored procedure SQL files
# Injected by backend when running from SaaS mode:
env['FORGE_STASH_DIR'] = '/path/to/stash/airops'

# Inside the scripts -- path resolution:
stash_dir = os.environ.get('FORGE_STASH_DIR')
if stash_dir:
    sql_path = os.path.join(stash_dir, 'data/func', filename)
else:
    sql_path = os.path.join('data/func', filename)  # local default

10. Challenge 6: Agent App Deployment -- DAB Preserved

DAB and app.yaml are Databricks platform standards. They MUST be preserved. This project needs internal validation -- going against the platform is not an option.

Two Layers

Resource creation (tables, functions, genie, KA, experiments) via SDK during wizard. Deployment (packaging agent + deploying as App) via DAB.

Target State

Setup App generates databricks.yml + app.yaml from .forge config. Deploys via DAB (Python SDK or CLI). Resource bindings, SP grants handled by DAB.

Deploy Flow (DAB-compatible)

StepActionMechanism

12. ConfigProvider Pattern

Interface (Implemented)

Both providers implement the same interface. JavaScript side in visual/backend/lib/config-provider.js.

// Full interface -- both providers implement all methods
interface ConfigProvider {
  list()            // returns all key-value pairs as object
  get(key)          // returns string value or undefined
  set(key, value)   // write one key
  setMany(pairs)    // write multiple keys atomically
  toggle(key)       // flip true/false string
  listByPrefix(pfx) // returns entries whose key starts with pfx
  toEnvDict()       // returns flat object for subprocess env injection
  deleteKey(key)    // remove a key
}

Implementations

ImplementationBacking StoreModesNotes
LocalConfigProvider .env.local file on disk Mode C (local dev) Wraps existing dotenv read/write. Zero behavior change for local mode.
ForgeConfigProvider .forge YAML held in-memory zip on UC Volume Modes A & B (SaaS) In-memory zip, event-driven flush to UC Volume on each meaningful write. setMany() used for multi-key saves to avoid multiple flushes.

ForgeConfigProvider: In-Memory Zip + Flush

On startup (SaaS mode), the backend downloads the .forge.zip from UC Volume once and holds it in process memory. All reads serve from memory -- zero network calls. Writes update the in-memory zip then flush to UC Volume (single w.files.upload(), ~200ms). On app restart, re-download from Volume restores last flushed state.

App start
→
Download .forge.zip (UC Volume)
→
Hold in memory (AdmZip)
→
set/setMany triggers flush
→
Upload back to UC Volume

Call Sites to Refactor

LayerCurrentCountAction
Node.js backend Direct .env.local file read/write 36 call sites Refactor to ConfigProvider.get() / .set()
Node.js backend (total refs) All .env.local references 98 refs Audit and migrate
Python files os.environ / dotenv reads 272 refs across 32 files ZERO changes needed -- subprocess env injection handles it

Migration Steps

  1. Phase 1 (pure refactor): Create LocalConfigProvider that wraps existing .env.local read/write functions. No behavior change. All 36 call sites switch to the abstraction.
  2. Phase 2 (add SaaS): Create ForgeConfigProvider that reads/writes .forge YAML. Mode detection at startup picks the right provider.
  3. Phase 3 (cleanup): Remove all direct .env.local references from the codebase.

Python Side: No Changes

// Node.js backend already injects env vars into subprocess calls:
const env = configProvider.to_env_dict();
spawn('python', [script], { env: { ...process.env, ...env } });

Python scripts read from os.environ as they do today. The ConfigProvider feeds the right values regardless of backing store.

The Problem

Before: user had npm, uv, Python 3.11+, Databricks CLI all installed locally. After: none of that exists on the user's machine. It ALL must be available inside the DBX App runtime.

Millimeter-by-Millimeter: What Breaks

mm R.1: uv not available

41 subprocess calls use uv run python. Fix options:

  • Option A: Install uv at startup: pip install uv && uv sync in app.yaml command. Cleanest. uv is pip-installable. uv sync creates the venv from pyproject.toml.
  • Option B: Replace ALL uv run python with just python (after pip-installing deps)
  • Option C: Create a wrapper script that tries uv run python, falls back to python
Wall: uv sync needs network access to download packages. Takes 30-60s. This happens on EVERY app restart, not just first deploy. Alternative: run uv sync once at deploy time, commit the .venv/ to workspace files. Then uv run just uses the existing venv -- fast.

mm R.2: Python packages not installed

Even if uv is installed, the venv needs to be created and populated. pyproject.toml has 20+ dependencies including large ones (mlflow, langchain, pandas, pyarrow). Full install takes 2-3 minutes.

Wall: If this happens on every restart, startup is painfully slow. Install once at deploy time, persist venv in workspace files. Or: use pip install -r requirements.txt in the app.yaml command with exact versions (fast pip resolve).

mm R.3: Databricks CLI (Go binary) not available

The Go binary is NOT pip-installable. It's a standalone binary.

In DBX App mode, do we even need it?

CLI CallNeeded in DBX App Mode?Reason
databricks auth profiles NO Auth comes from SP. No CLI profiles.
databricks auth login NO No interactive login needed.
databricks database list-database-instances YES Replace with GET /api/2.0/database/instances via SDK
databricks database get-database-instance YES Replace with GET /api/2.0/database/instances/{name} via SDK
databricks apps get YES Replace with GET /api/2.0/apps/{name} via SDK

Decision: Replace all databricks CLI calls with REST API equivalents. This removes the Go binary dependency entirely. 5 call sites in the backend. Each is a simple SDK/REST call.

mm R.4: Node.js / npm for Agent App build

Agent App startup: npm install && npm run build:client && npm run start. This requires Node.js + npm in the Agent App's DBX App runtime. The Agent App is a SEPARATE DBX App from the Setup App.

Current app.yaml starts with uv run python which then spawns Node as a subprocess. This already works today -- the Agent App is already deployed as a DBX App this way. Node.js IS available in the DBX App runtime.

mm R.5: npm for Setup App

Setup App backend is Node.js with pre-built frontend and committed node_modules. No npm install needed at runtime. Just Node.js to run the server.

mm R.6: Eliminate npm from Agent App runtime (PRE-BUILD)

Currently the Agent App builds the chat frontend at startup (npm install && npm run build:client). But the startup hook already skips the build if dist/ exists.

Solution: pre-build and include app/client/dist/ and app/server/dist/ in the deployment bundle. Remove client/dist/ from .databricksignore. Startup sees dist exists, skips npm. npm eliminated from Agent App runtime.

mm R.7: No rebuild needed when adding tools

Tools are Python -- no build step. The agent discovers tools at startup by scanning tools/ directory via pkgutil.iter_modules(). Adding a tool = drop a .py file + restart agent. The factories (ka_factory, api_factory, a2a_factory) discover tools from env vars at startup. The .forge tool specs are read at startup. Zero build step for any tool change. Just restart.

The PY Abstraction (Implemented)

const PY = { cmd, pre } is defined once at startup in visual/backend/index.js. All 40+ subprocess calls use it -- zero hardcoded uv run python strings remain.

// visual/backend/index.js -- resolved once at startup
const FORGE_MODE = process.env.FORGE_MODE === 'true';
const PY = FORGE_MODE
  ? { cmd: 'python', pre: [] }              // SaaS: system Python, no uv
  : { cmd: 'uv',    pre: ['run', 'python'] }; // Local: uv venv isolation

// Usage in every subprocess call:
spawn(PY.cmd, [...PY.pre, 'scripts/some_script.py'], { env });

Migration: 40+ subprocess calls migrated from spawn('uv', ['run', 'python', ...]) to spawn(PY.cmd, [...PY.pre, ...]). Consistent pattern, single change point.

Inch 4: Config Persistence -- Stateful Backend

In SaaS mode, the Node.js backend is stateful. It holds the project zip in process memory.

  1. User opens project → zip downloaded from UC Volume into memory
  2. User fiddles with wizard → changes accumulate in the in-memory zip
  3. Each meaningful write (config save, file gen, resource creation) → update in-memory zip + flush to UC Volume
  4. Page refresh → backend still alive, serves from memory. No data lost.
  5. App restart → re-download zip from Volume. Last flushed state restored.

Flush: event-driven, not periodic. User saves → flush. User idle → zero flushes. One w.files.upload(), <1MB, ~200ms.

Design shift: current backend is stateless (reads disk per request). SaaS backend is stateful (holds zip in memory). ConfigProvider abstraction handles both.
