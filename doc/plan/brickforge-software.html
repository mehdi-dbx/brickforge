1. Vision & Core Shift

Vision

BrickForge goes from a locally-cloned project into a cloud-native SaaS product. The user never clones a repo, never runs a terminal, never installs anything. They open a browser, create an agent, deploy it. Done.

The user also needs access to the generated code. They must be able to CI/CD it, edit it in an IDE, and own it. The Setup App is a PROJECT GENERATOR, not just a deployer.

The Core Shift

Before

User clones repo → has all code locally → runs scripts → deploys from local machine

After

User opens browser → Setup App does everything → code never touches user's machine (unless they want it to)

Setup App = The Product

The visual Setup App (currently visual/) IS the product. It:

  • Creates and manages .forge projects (multi-project: save/load/switch)
  • Walks users through domain setup via LLM-assisted wizard
  • Provisions UC data (tables, functions, procedures)
  • Deploys the Agent App to Databricks
  • Generates a git-ready project the user can own
  • Runs as a Databricks App itself (or locally for dev)

3 Runtime Modes

A: Hosted Maintainer deploys Setup App once, users open URL Quick start / demo Databricks SSO to user's workspace
B: User's DBX App User deploys Setup App to their own workspace Production use DBX App SP (user's permissions)
C: Local node visual/backend/index.js Dev / offline .env.local / CLI profile

3. .forge Schema

What It Is

  • Internal serialization format -- machine-generated, machine-consumed
  • One .forge file per project
  • User NEVER writes YAML by hand -- the Setup App wizard generates it
  • Contains BOTH domain content (tables, SQL, prompts) AND infrastructure config (schema, warehouse, genie IDs)

Domain Content

  • Domain description
  • Table definitions (DDL, seed data references)
  • UC function SQL (inline or sidecar files)
  • Stored procedure SQL
  • Tool definitions (declarative: 7 patterns)
  • System prompt + knowledge base + starter prompts
  • KA configurations
  • Eval dataset + scorer config
  • UI dashboard config

10. Challenge 6: Agent App Deployment -- DAB Preserved

DAB and app.yaml are Databricks platform standards. They MUST be preserved. This project needs internal validation -- going against the platform is not an option.

Two Layers

Resource creation (tables, functions, genie, KA, experiments) via SDK during wizard. Deployment (packaging agent + deploying as App) via DAB.

Target State

Setup App generates databricks.yml + app.yaml from .forge config. Deploys via DAB (Python SDK or CLI). Resource bindings, SP grants handled by DAB.

Deploy Flow (DAB-compatible)

StepActionMechanism
1 Provision resources (tables, functions, genie, KA, experiment) SDK calls during wizard -- already done before deploy
2 Generate databricks.yml from .forge Template + config injection (resources, targets, app name, bindings)
3 Generate app.yaml from .forge Template + config injection (startup command, env vars)
4 DAB deploy databricks bundle deploy (CLI) or Python SDK w.apps.deploy()
5 Resource bindings applied Automatic by DAB (experiment, warehouse, genie, endpoint bound to app SP)
6 Post-deploy grants SDK grant calls

Why DAB for Deployment

  • Databricks platform standard -- required for internal validation
  • Resource binding is declarative in databricks.yml
  • SP permissions auto-applied via resource bindings
  • Workspace state tracking (.databricks/bundle/)
  • Reproducible: same databricks.yml = same deployment

Stash Includes Bundle Configs

app.yaml and databricks.yml are project artifacts, part of the stash. Setup App generates project-specific versions by injecting .forge config values into templates at deploy time.

12. ConfigProvider Pattern

Interface

class ConfigProvider:
    def get(key) → str | None
    def set(key, value) → void
    def list() → dict
    def toggle(key) → void
    def to_env_dict() → dict   # for subprocess injection

Implementations

ImplementationBacking StoreModes
LocalConfigProvider .env.local file Mode C (local dev)
ForgeConfigProvider .forge YAML on UC Volume Modes A & B (SaaS)

Call Sites to Refactor

LayerCurrentCountAction
Node.js backend Direct .env.local file read/write 36 call sites Refactor to ConfigProvider.get() / .set()
Node.js backend (total refs) All .env.local references 98 refs Audit and migrate
Python files os.environ / dotenv reads 272 refs across 32 files ZERO changes needed -- subprocess env injection handles it

Migration Steps

  1. Phase 1 (pure refactor): Create LocalConfigProvider that wraps existing .env.local read/write functions. No behavior change. All 36 call sites switch to the abstraction.
  2. Phase 2 (add SaaS): Create ForgeConfigProvider that reads/writes .forge YAML. Mode detection at startup picks the right provider.
  3. Phase 3 (cleanup): Remove all direct .env.local references from the codebase.

Python Side: No Changes

// Node.js backend already injects env vars into subprocess calls:
const env = configProvider.to_env_dict();
spawn('python', [script], { env: { ...process.env, ...env } });

Python scripts read from os.environ as they do today. The ConfigProvider feeds the right values regardless of backing store.

Millimeter-by-Millimeter: What Breaks

mm R.1: uv not available

41 subprocess calls use uv run python. Fix options:

  • Option A: Install uv at startup: pip install uv && uv sync in app.yaml command. Cleanest. uv is pip-installable. uv sync creates the venv from pyproject.toml.
  • Option B: Replace ALL uv run python with just python (after pip-installing deps)
  • Option C: Create a wrapper script that tries uv run python, falls back to python
Wall: uv sync needs network access to download packages. Takes 30-60s. This happens on EVERY app restart, not just first deploy. Alternative: run uv sync once at deploy time, commit the .venv/ to workspace files. Then uv run just uses the existing venv -- fast.

mm R.2: Python packages not installed

Even if uv is installed, the venv needs to be created and populated. pyproject.toml has 20+ dependencies including large ones (mlflow, langchain, pandas, pyarrow). Full install takes 2-3 minutes.

Wall: If this happens on every restart, startup is painfully slow. Install once at deploy time, persist venv in workspace files. Or: use pip install -r requirements.txt in the app.yaml command with exact versions (fast pip resolve).

mm R.3: Databricks CLI (Go binary) not available

The Go binary is NOT pip-installable. It's a standalone binary.

In DBX App mode, do we even need it?

CLI CallNeeded in DBX App Mode?Reason
databricks auth profiles NO Auth comes from SP. No CLI profiles.
databricks auth login NO No interactive login needed.
databricks database list-database-instances YES Replace with GET /api/2.0/database/instances via SDK
databricks database get-database-instance YES Replace with GET /api/2.0/database/instances/{name} via SDK
databricks apps get YES Replace with GET /api/2.0/apps/{name} via SDK

Decision: Replace all databricks CLI calls with REST API equivalents. This removes the Go binary dependency entirely. 5 call sites in the backend. Each is a simple SDK/REST call.

mm R.4: Node.js / npm for Agent App build

Agent App startup: npm install && npm run build:client && npm run start. This requires Node.js + npm in the Agent App's DBX App runtime. The Agent App is a SEPARATE DBX App from the Setup App.

Current app.yaml starts with uv run python which then spawns Node as a subprocess. This already works today -- the Agent App is already deployed as a DBX App this way. Node.js IS available in the DBX App runtime.

mm R.5: npm for Setup App

Setup App backend is Node.js with pre-built frontend and committed node_modules. No npm install needed at runtime. Just Node.js to run the server.

mm R.6: Agent App's startup chain in DBX App mode

Today this ALREADY WORKS -- the current Agent App deploys this way. The only new thing: the Setup App triggers the deploy, instead of databricks bundle deploy from a terminal. No change needed for the Agent App's runtime.

The pythonCmd() Abstraction

// Detect mode, use appropriate Python command
function pythonCmd() {
  if (isDbxAppMode()) {
    return ['python'];           // system Python, pip-installed deps
  }
  if (uvAvailable()) {
    return ['uv', 'run', 'python']; // venv isolation
  }
  return ['python'];               // fallback
}

Startup Command for Setup App

# Setup App app.yaml
command:
  - "pip install uv && uv sync && node visual/backend/index.js"

This: (1) Installs uv (fast, ~5s), (2) Creates venv and installs Python deps (slow first time, ~2-3 min; fast if .venv persisted), (3) Starts the Node.js Setup App server.

Startup Optimization Options

OptionHowTradeoff
A Run uv sync at deploy time, persist .venv/ in workspace files Fast restart, more deploy complexity
B Use pip install -r requirements.txt instead (no uv needed at all for Setup App) Simplest, but then backend still calls uv run python...
C Pre-build a Docker layer with deps (if DBX Apps support custom images) Best perf, most setup

mm D1.3: build-release.sh

#!/bin/bash
VERSION=${1:-"dev"}
OUT="brickforge-${VERSION}"

# Create clean directory
rm -rf "$OUT" "$OUT.tar.gz"
mkdir -p "$OUT"

# Copy files (exclude bloat)
rsync -a --exclude='.git' --exclude='.venv' --exclude='node_modules' \
         --exclude='__pycache__' --exclude='.mypy_cache' \
         --exclude='.DS_Store' --exclude='.claude' \
         --exclude='.env.local' --exclude='.env.*.local' \
         ./ "$OUT/"

# Include ONLY the visual backend node_modules (required, pre-committed)
cp -r visual/backend/node_modules "$OUT/visual/backend/"

# Include pre-built frontend dist
cp -r visual/frontend/dist "$OUT/visual/frontend/"

# Compress
tar czf "$OUT.tar.gz" "$OUT"
echo "Built: $OUT.tar.gz ($(du -h "$OUT.tar.gz" | cut -f1))"

mm D1.4: start.sh launcher

#!/bin/bash
set -e

echo "BrickForge Setup App"
echo "===================="

# Check Node.js
if ! command -v node &>/dev/null; then
  echo "ERROR: Node.js not found. Install from https://nodejs.org/"
  exit 1
fi

# Check Python
PYTHON=""
for cmd in python3 python; do
  if command -v "$cmd" &>/dev/null; then
    PYTHON="$cmd"
    break
  fi
done
if [ -z "$PYTHON" ]; then
  echo "ERROR: Python 3.11+ not found. Install from https://python.org/"
  exit 1
fi

# Install Python deps (first run only)
if [ ! -f ".deps_installed" ]; then
  echo "Installing Python dependencies (first run)..."
  $PYTHON -m pip install -r requirements.txt --quiet
  touch .deps_installed
fi

# Start
echo "Starting BrickForge at http://localhost:9000"
node visual/backend/index.js

mm D1.6: The uv problem in start.sh

start.sh installs deps with pip. But 41 subprocess calls in the backend use uv run python. If uv isn't installed, those fail.

Best answer: The pythonCmd() abstraction. In local mode without uv, it uses python. With uv, it uses uv run python. The release archive runs without uv.

This means the release archive doesn't need uv at all. Just Python + pip + Node.js.

mm D2.2: The problem with pip

pip packages are designed for Python libraries, not multi-runtime applications. Including 8MB of Node.js files as package data works but is unusual. The user STILL needs Node.js installed separately. pip can't install Node.js.

mm D2.3: Publishing to PyPI

  1. Check name availability: pip index versions brickforge or visit pypi.org/project/brickforge/ (404 = available)
  2. Create account at pypi.org, get API token
  3. Build: python -m build (creates dist/brickforge-1.0.0.tar.gz)
  4. Upload: twine upload dist/*
  5. No approval process -- upload and it's live immediately
  6. Then anyone can: pip install brickforge && brickforge

Decision: pip install is a STRETCH GOAL. GitHub Release first. If demand exists, add pip later.

mm D3.2: What deploy-setup actually does

  1. Authenticate: databricks auth login --host <workspace> (opens browser)
  2. Create the DBX App: POST /api/2.0/apps with name brickforge-setup
  3. Upload source files to workspace: files API
  4. Create deployment: POST /api/2.0/apps/{name}/deployments
  5. Wait for app to start
  6. Print URL

mm D3.3: What files get uploaded

Same as the release archive, but uploaded to workspace files instead of extracted locally. The upload script walks the file tree and PUTs each file.

mm D3.4: Auth for deploy-setup

User needs a Databricks PAT or CLI profile. The script uses the SDK which reads from: DATABRICKS_HOST + DATABRICKS_TOKEN env vars, or ~/.databrickscfg CLI profile, or interactive databricks auth login. No new auth mechanism needed.

mm D4.2: Pros/cons

Pros: zero prereqs (just Docker), works everywhere, reproducible. Cons: Docker Desktop required (not everyone has it), ~500MB image size, overkill for a dev tool.

Decision: Nice-to-have. Build the Dockerfile but don't make it the primary path.

mm W.2: Path separators

All Python code uses pathlib.Path or os.path -- cross-platform by default. The Node.js backend uses path.resolve() -- cross-platform. No hardcoded / path separators in critical code.

mm W.3: The uv question on Windows

uv works on Windows (pip-installable). But with the pythonCmd() abstraction, uv becomes optional on all platforms. Users who have it get venv isolation; users who don't just use system Python.

What NOT to Do

  • Don't publish to npm (Python bundling nightmare)
  • Don't build Electron/Tauri (desktop wrapper for a web server -- wrong paradigm)
  • Don't build platform installers (MSI, DMG, deb -- maintenance hell)
  • Don't build Homebrew formula (Mac-only, another package to maintain)
  • Don't build snap/flatpak (Linux-only, containerization overhead)
  • Don't build a custom update mechanism (just release new versions on GitHub)
