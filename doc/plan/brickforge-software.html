Slide 3 -- The .forge File

One File.
All Config.

Internal serialization format -- machine-generated, machine-consumed. One .forge file per project. User NEVER writes YAML by hand. The Setup App wizard generates it.

Contains BOTH domain content (tables, SQL, prompts) AND infrastructure config. Replaces .env.local entirely in SaaS mode.

Storage

/Volumes/{catalog}/{schema}/brickforge/projects/{name}/

Sidecar files (CSV, SQL, PDFs) live alongside with relative paths.

First Example

stash/airops/airops.forge -- extracted from the original airops domain.

Slide 6 -- ConfigProvider

One Interface.
Two Backends.

The ConfigProvider abstraction lets the entire codebase work identically in local mode and SaaS mode. The Python side needs ZERO changes.

Interface Methods

get(key) set(key, value) list() toggle(key) to_env_dict()

Migration Strategy

Pure refactor first: LocalConfigProvider wraps existing functions. Then add ForgeConfigProvider. No behavior change until the switch.

The pythonCmd() Abstraction

Detect mode, use appropriate command. Returns ['uv', 'run', 'python'] locally or ['python'] in DBX App mode. Same pattern as ConfigProvider.

Slide 8 -- Packaging

4 Delivery Paths

From GitHub Release to Docker. Clean repo is 25MB, compressed ~12MB. The two-runtime problem (Node + Python) means no single package manager.

Priority 1

GitHub Release

tar.gz/zip archive. Prereqs: Node.js + Python. build-release.sh script excludes bloat, includes pre-built frontend and committed node_modules.

User flow: curl -L .../brickforge.tar.gz | tar xz && cd brickforge-* && ./start.sh

Priority 2

deploy-setup

One-command DBX App deploy. python deploy-setup.py authenticates, uploads source, creates app, prints URL. Prereqs: Python + Databricks auth.

Priority 3

Docker

Zero prereqs (just Docker). FROM node:20-slim + Python. ~500MB image. Overkill for dev but reproducible.

Stretch

pip install brickforge

Marginal over GitHub Release. pip can't install Node.js. User still needs Node separately. Build if demand exists.

What Goes in the Archive

brickforge-v1.0.0/ visual/ backend/ index.js, lib/, package.json node_modules/ # 8MB, required frontend/ dist/ # 580KB, pre-built agent/ # 5 Python files, ~30KB app/ client/src/ # 528KB React server/src/ # 60KB Express packages/ # 38MB shared tools/ # framework, ~20KB data/ # init, gen, py conf/ # skeleton prompts deploy/ # deploy pipeline stash/ # airops example pyproject.toml, requirements.txt start.sh / start.bat setup-app.yaml

Slide 11 -- Execution Roadmap

Inch by Inch

Start from where we are. Take one step. Hit the wall. Solve the wall. Next step. 8 inches from current state to multi-project SaaS.

1

Setup App as DBX App (Mode B)

Create app.yaml for Setup App. Upload entire repo. pip install uv && uv sync && node visual/backend/index.js

No dependencies -- starting point
2

Port + Auth

process.env.PORT || 9000 -- one line. Mode B auth via app's SP with user permissions.

Depends on: Inch 1
3

First Deploy Test

Deploy, open URL, walk through setup steps. Fix what breaks -- uv, Node version, CORS, etc.

Depends on: Inch 2
4

Config Persistence

.env.local is ephemeral in DBX App mode. Short-term: reconfigure on restart. Long-term: ConfigProvider reads .forge from UC Volume.

Depends on: Inch 3 (working deploy)
5

DAB-less Agent Deploy

Replace deploy.sh (DAB CLI) with deploy_via_api.py (REST API). Upload source to workspace files, generate app.yaml, call Apps API.

Depends on: Inch 3 (verified REST API endpoints)
6

Agent App File Manifest

Define exactly which files are Agent App vs Setup App. Agent gets: agent/, app/, tools/, data/, conf/, eval/.

Depends on: Inch 5
7

.forge Config Injection

Setup App generates Agent App app.yaml with all env vars from .forge. Domain files written to workspace before deploy.

Depends on: Inch 4 + Inch 6
8

Multi-Project

Save/load/switch .forge projects on UC Volume. Per-project schema isolation. Separate Agent App instances, shared Setup App.

Depends on: Inch 7 (full pipeline working)
