# BrickForge SaaS Transition -- Engineering Details

> The hard stuff. What actually needs to change and why.
> Companion to saas-master-plan.md.
> NEVER DELETE OR OVERWRITE THIS FILE.

## Resolved Design Decisions

| Question | Answer |
|----------|--------|
| Hosted mode auth | Databricks SSO -- user authenticates via SSO to their workspace |
| Can DBX App deploy another DBX App? | Yes -- it's just Python code executing from App, calls Apps REST API |
| Local mode | Still supported -- `node visual/backend/index.js`, uses `.env.local` |
| dbc/ courseware | Separated to `forge-dbc` branch -- different project entirely |
| `.forge` storage | UC Volume on user's workspace |
| User writes YAML? | NEVER -- wizard generates it, user only sees UI |
| Tools need custom Python? | NO -- 3 declarative patterns, framework generates code at runtime |
| Notebooks? | STRICTLY PROHIBITED -- code-first only |

---

## The Core Shift

**BEFORE:** User clones repo -> has all code locally -> runs scripts -> deploys from local machine
**AFTER:** User opens browser -> Setup App does everything -> code never touches user's machine

This means EVERY piece of logic that currently runs locally via `uv run python ...` or `bash ...`
must either:
1. Move INTO the Setup App backend (Node.js)
2. Execute REMOTELY on Databricks via SDK/API calls
3. Both

## Challenge 1: Where Does the Code Live?

### Current state
```
User's machine:
  agent-forge/           <-- cloned repo
    agent/               <-- Python agent runtime
    app/                 <-- React + Express chat app
    tools/               <-- Python tool files
    data/                <-- SQL, CSV files
    deploy/              <-- deploy.sh, sync scripts
    visual/              <-- Setup App (Node.js)
    conf/                <-- prompts, KA configs
```
Everything is local. Deploy pushes it to Databricks.

### Target state
```
Setup App (running as DBX App or locally):
  visual/                <-- The Setup App itself (what the user sees)
  _bundle/               <-- Embedded copy of agent code (static artifact)
    agent/
    app/
    tools/               <-- empty, populated from .forge at deploy time
    data/                <-- empty, populated from .forge at deploy time
    conf/                <-- empty, populated from .forge at deploy time

User's Databricks workspace:
  UC Volume:
    brickforge/projects/
      airops.forge       <-- project 1
      retail.forge       <-- project 2
  Workspace files:
    /Workspace/Users/{user}/brickforge-{project}/
      agent/             <-- uploaded by Setup App
      app/               <-- uploaded by Setup App
      tools/             <-- generated from .forge by Setup App
      data/              <-- generated from .forge by Setup App
      conf/              <-- generated from .forge by Setup App
      app.yaml           <-- generated by Setup App
      databricks.yml     <-- generated by Setup App
  Databricks Apps:
    brickforge-airops    <-- deployed Agent App
    brickforge-retail    <-- deployed Agent App
```

### How code gets from Setup App to Agent App
1. Setup App carries a `_bundle/` directory with the framework code (agent runtime, React app, Express server)
2. This bundle is the SAME for every project -- it's the engine
3. On "deploy", Setup App:
   a. Reads the `.forge` file for the project
   b. GENERATES domain-specific files from `.forge`:
      - `tools/*.py` -- Python tool files generated from declarative specs
      - `data/default/func/*.sql` -- UC function SQL from `.forge`
      - `data/default/proc/*.sql` -- stored procedure SQL from `.forge`
      - `data/default/csv/*.csv` -- seed data from `.forge`
      - `data/default/init/*.sql` -- DDL SQL from `.forge`
      - `conf/prompt/main.prompt` -- system prompt from `.forge`
      - `conf/prompt/knowledge.base` -- knowledge base from `.forge`
      - `conf/ka/*.yml` -- KA configs from `.forge`
   c. Uploads bundle + generated files to workspace files
   d. Generates `app.yaml` and `databricks.yml` for this project
   e. Calls Databricks Apps API to create/deploy

### Critical insight
The `.forge` file must contain EVERYTHING needed to generate the domain files.
Not references to files -- the ACTUAL CONTENT.
SQL is inline in the YAML. Prompts are inline. Tool specs are inline.
CSVs might be too large for inline -- those go to UC Volumes as sidecar uploads.

## Challenge 2: The Setup App Backend Shift

### Current state
The Setup App backend (`visual/backend/index.js`) does everything by spawning local Python:
```javascript
runCommand('uv', ['run', 'python', 'data/init/create_all_assets.py'])
runCommand('uv', ['run', 'python', '-c', SAVE_SCHEMA_SCRIPT])
runCommand('bash', ['deploy/deploy.sh'])
```

It assumes:
- `uv` is installed locally
- Python venv exists locally
- All scripts exist at relative paths
- `.env.local` is a local file

### Target state
When running as a DBX App, there is NO local Python, NO uv, NO bash.
The backend must either:

**Option A: Rewrite in pure Node.js + Databricks SDK (REST API)**
- All data provisioning via Databricks REST API / SQL Statement API
- No Python subprocess calls
- Clean but massive rewrite

**Option B: Keep Python but run it differently**
- Bundle a Python runtime in the DBX App
- Or: execute Python via Databricks Jobs API (serverless)
- Or: use Databricks Serverless Compute to run Python scripts remotely

**Option C: Hybrid**
- Simple operations (read env, save file, list resources) -> Node.js + REST API
- Complex operations (create all assets, deploy) -> submitted as Databricks jobs
- This is the most pragmatic path

### What the backend currently does via Python subprocess:
1. `create_all_assets.py` -- creates schema, tables, functions, procedures, genie space
2. `create_genie_space.py` -- creates genie space
3. `create_kas_from_yml.py` -- creates KA endpoints
4. `setup_dbx_env.py` -- interactive CLI setup (NOT needed in visual app)
5. `deploy/deploy.sh` -- full deploy pipeline
6. Various inline Python scripts for save/test operations
7. `reset_workspace.py` -- cleanup

Each of these needs a migration strategy.

## Challenge 3: .env.local Doesn't Exist in DBX App Mode

### Current state
Everything reads from `.env.local` -- a local file.
The Setup App reads it, writes it, passes it to subprocesses.

### Target state
In DBX App mode, there's no local filesystem in the traditional sense.
Options:
- Store config in a **UC Volume** as a JSON/YAML file per project
- Store config in **app environment variables** (limited, not per-project)
- Store config in **Databricks Secrets** (for sensitive values)
- Store config in **workspace files** alongside the project

### Design choice
The `.forge` file IS the config. It replaces `.env.local` for project-specific config.
Global config (host, token, warehouse) comes from the Databricks App's own auth context.
Per-project config (schema, genie spaces, KA endpoints) lives in `.forge`.

When running locally (mode C), `.env.local` still works as a convenience.
The backend abstracts over both: "give me the config for this project" works
regardless of whether it's reading `.env.local` or a `.forge` from UC Volume.

## Challenge 4: Tool Generation at Runtime

### Current state
Tools are Python files in `tools/`. The agent discovers them via `pkgutil`.
Each tool is a `@tool`-decorated function with hardcoded SQL or procedure calls.

### Target state
Tools are DEFINED in `.forge` as declarative specs:
```yaml
tools:
  - name: query_inventory
    pattern: sql_read
    function: get_inventory_by_warehouse
    params: [{name: warehouse_id, type: str}]
    description: "Query current inventory levels by warehouse"
```

The framework GENERATES the Python tool code at agent startup from these specs.
We already have patterns for this:
- `ka_factory.py` generates KA tools from `PROJECT_KA_*` env vars
- The same factory pattern extends to SQL read and action tools

### What needs to be built
A `tool_factory.py` that reads `.forge` tool specs and generates `@tool` functions:
- SQL read pattern: substitute params into function SQL, execute via warehouse
- Action pattern: call stored procedure with params
- KA pattern: already done in `ka_factory.py`

This eliminates the need for tool Python files entirely.
The `tools/` directory becomes empty in the clean framework.
Domain tools are generated at runtime from `.forge`.

## Challenge 5: Data Provisioning Without Local Scripts

### Current state
`create_all_assets.py` runs locally:
1. Creates UC schema
2. Reads CSV files from `data/default/csv/`
3. Reads DDL from `data/default/init/`
4. Executes SQL via Databricks SDK
5. Creates functions from `data/default/func/`
6. Creates procedures from `data/default/proc/`
7. Creates Genie space

### Target state
The Setup App does all of this via Databricks REST API / SQL Statement API:
1. Schema creation: `POST /api/2.0/unity-catalog/schemas`
2. Table creation: SQL Statement API (`POST /api/2.0/sql/statements`)
3. CSV upload: write to UC Volume, then `COPY INTO` or `INSERT`
4. Functions: SQL Statement API
5. Procedures: SQL Statement API
6. Genie space: Genie REST API

All SQL comes from the `.forge` file (inline content).
All CSV data comes from `.forge` (inline rows) or UC Volume (uploaded sidecar files).

### Migration path
1. First: make `create_all_assets.py` work with `.forge` as input (instead of scanning dirs)
2. Then: port the SDK calls to Node.js REST API calls in the Setup App backend
3. Or: submit `create_all_assets.py` as a Databricks job (hybrid approach)

## Challenge 6: Agent App Deployment Without DAB CLI

### Current state
`deploy.sh` uses `databricks bundle deploy` (DAB CLI) which:
1. Validates databricks.yml
2. Uploads source code
3. Creates/updates Databricks App resource
4. Binds experiment, genie space, warehouse
5. Starts the app
6. Runs grant scripts

This requires the Databricks CLI installed locally.

### Target state
The Setup App deploys via REST API:
1. Upload source files to workspace: `POST /api/2.0/workspace/import`
2. Create app: `POST /api/2.0/apps`
3. Deploy app: `POST /api/2.0/apps/{name}/deployments`
4. Grant permissions: UC REST API calls
5. No CLI needed

### Alternative: DAB-less deployment
Databricks Apps can be deployed without DAB:
1. Upload code to workspace files
2. Create `app.yaml` with startup command
3. Call Apps API directly
4. The `databricks.yml` bundle config is only needed for DAB -- we can skip it entirely

This is actually SIMPLER than the current DAB-based approach.
The Setup App generates `app.yaml` and uploads code. Done.

## Challenge 7: The Prompt/Knowledge Base

### Current state
`conf/prompt/main.prompt` is a file on disk. The agent reads it at startup.

### Target state
The system prompt is in `.forge` (inline). On deploy, it's written to
the agent's config directory in workspace files. The agent reads it
the same way -- it doesn't care where the file came from.

Same for `knowledge.base` and `user.prompt`.

## Execution: Inch by Inch

### Approach
DO NOT try to architect the whole system upfront.
Start from where we are. Take one step. Hit the wall. Solve the wall. Take the next step.
Every inch reveals the next inch.

---

### Inch 1: Make Setup App a deployable DBX App (Mode B)

**Starting point:** `node visual/backend/index.js` runs locally.

**What I need:**
- A `visual/app.yaml` for the Setup App
- Startup command: install Python deps (the backend spawns `uv run python`), then start Node
- Something like: `pip install -r requirements.txt && node backend/index.js`

**First wall:** The backend uses `PROJECT_ROOT = path.resolve(__dirname, '../..')` to find agent code, data dirs, scripts. When deployed as a DBX App, the ENTIRE REPO is uploaded. So the paths still work -- the relative structure is preserved.

**Key insight:** For Mode B, if I upload the ENTIRE repo as the Setup App source, all subprocess calls still work. Python scripts are there. SQL files are there. The `.env.local` path is relative and still resolves. The only thing that changes is WHERE the process runs (Databricks instead of laptop).

**Deliverable:**
1. Create `visual/app.yaml` (or root-level `setup-app.yaml`)
2. Startup command installs Python deps + starts Node
3. Test: deploy to a workspace, open in browser, verify Setup tab works

**What can break:**
- `uv` might not be available in DBX App runtime -> use `pip` instead
- Node.js version compatibility
- `.env.local` won't exist on first deploy -> Setup App must handle missing `.env.local` gracefully (it already does for most steps)
- Port: DBX Apps expect the app to listen on a specific port (usually 8080 or from env var `PORT`)

**Verification:** Deploy, open URL, click through Setup steps, verify tests pass.

---

### Inch 2: Handle the port and auth in DBX App mode

**Wall from Inch 1:** DBX Apps inject `PORT` env var. The Setup App currently hardcodes port 9000.

**Fix:** `const PORT = process.env.PORT || process.env.VISUAL_PORT || 9000`
Already partially there. Just ensure `PORT` env var is respected.

**Auth wall:** When running as a DBX App, the user is authenticated via Databricks SSO. The app gets user identity headers. The Databricks SDK uses the app's service principal for API calls, but the USER is the one who deployed the app and has permissions.

For Mode B (user's own workspace), the app's SP inherits permissions from the user who deployed it. The Setup App's API calls (create schema, create tables, etc.) run with the SP's permissions. The user needs to grant the SP access to their catalog/warehouse.

**This is the same pattern as the Agent App deployment.** The grant scripts already handle this.

**Deliverable:** Setup App respects `PORT` env var, auth works via DBX App SP.

### Inch 4: .env.local bootstrap in DBX App mode

**Wall:** On first deploy, there's no `.env.local`. The Setup App can't read config.

**But wait:** In Mode B, the Setup App IS in the user's workspace. The user walks through the Setup wizard which CREATES the `.env.local`. This is the SAME flow as today -- the wizard populates config step by step.

The only difference: the user opens a DBX App URL instead of `localhost:9000`.

**Fix needed:** The Setup App backend must handle missing `.env.local` gracefully at startup (not crash). It already mostly does this -- the Setup tab shows "not configured" for missing values.

But some startup code might fail if `.env.local` is missing. Need to audit:
- `require('dotenv').config({ path: ... })` -- this is fine, dotenv silently ignores missing files
- `parseEnvFile()` -- returns empty array if file missing (already handled)
- Subprocess calls that do `load_dotenv('.env.local')` -- dotenv handles missing files

**Likely OK.** But need to test.

### Inch 5: The Deploy button -- Setup App deploys Agent App

**Starting point:** Deploy button calls `bash deploy/deploy.sh`. The script uses `databricks bundle deploy` (DAB CLI).

**Wall:** In DBX App mode, `databricks` CLI might not be installed. And DAB requires a local `databricks.yml`.

**Two options:**
A. Ensure `databricks` CLI is available in the DBX App runtime (install via pip: `pip install databricks-cli`)
B. Replace DAB with direct REST API calls

**Option A is simpler for now.** The `databricks` CLI is a Python package. `pip install databricks-cli` makes it available. Then `deploy.sh` works as-is.

**But wait:** `deploy.sh` uses `databricks bundle deploy` which is part of the Databricks CLI v2 (the Go binary), NOT the Python `databricks-cli` package. The Go binary would need to be downloaded separately.

**Hmm.** So Option A might not work easily. The Go binary isn't pip-installable.

**Option B then:** Replace `deploy.sh` with a Python script that calls REST API:
1. Upload source to workspace files (`/api/2.0/workspace/import`)
2. Create app (`POST /api/2.0/apps`)
3. Create deployment (`POST /api/2.0/apps/{name}/deployments`)
4. Run grants (SDK calls, already in Python)

This is the DAB-less deployment. Actually simpler.

**Deliverable:** New `deploy/deploy_via_api.py` that does the same as `deploy.sh` but via REST API.

### Inch 6: What gets uploaded for the Agent App?

**Wall:** When the Setup App deploys the Agent App, what files go up?

**The Agent App needs:**
- `agent/` -- Python agent runtime (agent.py, start_server.py, genie_capture.py, utils.py)
- `app/` -- React frontend + Express server (the chat UI)
- `tools/` -- framework tools (sql_executor.py, ka_factory.py, get_current_time.py) + any domain tools loaded from .forge
- `data/` -- only the domain data files (populated from .forge)
- `conf/` -- prompts, KA configs (populated from .forge)
- `eval/` -- framework eval scripts (optional)
- `pyproject.toml` + `requirements.txt` -- Python deps
- `app.yaml` -- generated for this specific project (schema, warehouse, env vars)

**What does NOT go up:**
- `visual/` -- that's the Setup App, not the Agent App
- `stash/` -- domain packs stay on the Setup App side
- `deploy/` -- deploy scripts are Setup App side
- `scripts/` -- setup scripts are Setup App side
- `.env.local` -- config is baked into app.yaml env vars
- `doc/`, `edu/`, `.claude/` -- irrelevant

**Key insight:** The Setup App needs to know which files are "Agent App files" vs "Setup App files". This is a manifest or a well-defined split.

### Inch 7: The .forge config injection

**Wall:** The Agent App needs domain-specific config at runtime. Currently it reads `.env.local`. In DBX App mode, config comes from `app.yaml` env vars.

**The Setup App generates `app.yaml` for the Agent App with:**
```yaml
command:
  - "pip install -r requirements.txt && npm install && npm run build:client && npm run start"
env:
  - name: PROJECT_UNITY_CATALOG_SCHEMA
    value: "{from .forge}"
  - name: DATABRICKS_WAREHOUSE_ID
    value: "{from .forge}"
  - name: AGENT_MODEL_ENDPOINT
    value: "{from .forge}"
  - name: PROJECT_GENIE_{slug}
    value: "{from .forge}"
  - name: PROJECT_KA_{slug}
    value: "{from .forge}"
```

**The domain files (prompts, SQL, tools) are written to disk** in the uploaded workspace files before deployment. The agent reads them at startup from relative paths -- same as today.
