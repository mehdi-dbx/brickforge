## What this project is

Agent Forge is a Databricks-native framework and accelerator for building production-ready agentic AI solutions. It ships a reference flight-operations AI assistant (chat with an agent that monitors real-time flight risk, queries data via Genie MCP, and takes operational actions) plus an EU Passenger Rights Knowledge Assistant backed by regulatory PDFs. The framework is designed to be forked and adapted to any domain.

**Data source flags** (in `.env.local`):
- `USE_DEFAULT_DATA=true|false` -- include data/default/ tables (default: true)

## Synthetic data generation

The visual app's Data tab has a "generate" mode with a multi-step wizard:
1. **Domain**: describe your use case in natural language
2. **Schema**: LLM generates table schemas, user reviews/edits (names, columns, types, row counts)
3. **Data**: per-table row generation with preview, approve/reject/regenerate cycle
4. **Provision**: set target `PROJECT_UNITY_CATALOG_SCHEMA`, create tables in Databricks

Key implementation details:
- GenTerminal component uses `triggerKey` counter (not boolean) to avoid stale closure bugs
- Wizard state persisted to `data/gen/wizard-state.json` -- survives page refresh
- "Start fresh" button clears wizard state + generated files
- Provision runs `--mode=provision-gen` (only generated table SQL, not full create_all_assets.py)
- Generated files isolated in `data/gen/csv/` + `data/gen/init/` -- never mix with defaults

Backend routes (`visual/backend/index.js`):
- `GET /api/gen/status` -- model config + manifest + data source flags
- `GET /api/gen/tables` -- dynamic table discovery with `source: 'default' | 'generated'` tags
- `POST /api/gen/schema` -- SSE stream, LLM schema generation
- `POST /api/gen/data` -- SSE stream, LLM row generation
- `POST /api/gen/save` -- SSE stream, write CSV + SQL
- `POST /api/gen/provision` -- SSE stream, provision generated tables only
- `GET/PUT/DELETE /api/gen/wizard-state` -- wizard persistence
- `DELETE /api/gen/clear` -- remove all generated files

**Cleanup routes:**
- `GET /api/cleanup/resources` -- list deletable resources, validated against live workspace (only existing resources returned)
- `POST /api/cleanup/exec` -- SSE stream, deletes selected resources by ID

## Deployment

1. Configure: `./run setup` (interactive, writes `.env.local`)
2. Provision data: `uv run python data/init/create_all_assets.py`
3. Deploy: `./run deploy` (validates, syncs config, bundles, deploys, grants permissions)

The React frontend is built remotely at app startup (`npm install && npm run build:client`). First startup takes ~1-2 min; subsequent restarts are fast.

Deployment uses Databricks Asset Bundles (DAB). The bundle config (`databricks.yml`) is auto-synced from `.env.local` by `deploy/sync_databricks_yml_from_env.py`.

## Known issues / gotchas

1. **Stale PAT after workspace switch** -- regenerate token for new workspace; `/forge-setup-auth` handles this
2. **`create_all_assets.py` respects data flags** -- set `USE_DEFAULT_DATA` / `USE_GEN_DATA` in `.env.local` to control which tables are provisioned
3. **`databricks bundle validate` wrong profile** -- source `.env.local` before running CLI commands manually
4. **Grants timing on first deploy** -- `deploy.sh` may exit non-zero even though app is RUNNING; re-run `--check` after app is up
5. **"App already exists" on first deploy** -- deploy.sh has retry logic that auto-recovers
6. **Bundle lock from interrupted deploy** -- use `--force-lock`
7. **Frontend 500 on GET /** -- fixed; `start_server.py` now builds frontend remotely on first startup if `client/dist/` is absent
8. **FM workspace flavor mismatch** -- setup warns if Azure vs AWS workspace flavor doesn't match, which can cause IP ACL blocks
9. **VIRTUAL_ENV warning during data gen** -- harmless `uv` warning about mismatched venv; filtered from GenTerminal output
10. **Node.js v24 `req.on('close')` kills spawned processes** -- fixed: use `res.on('close')` with a `finished` guard in `runCommand()` (`visual/backend/index.js`). The `req` close event fires when the POST body is consumed, not on client disconnect.
11. **Setup auto-test caching** -- test results are cached in `SetupView` state, passed to `SetupDrawer` via `testCache` prop. Cache invalidated on reconfigure or exec completion. Avoids re-testing on every step click.
12. **"create all assets" requires schema confirmation** -- action `exec-assets` is in `NEEDS_CONFIGURE`, shows catalog.schema input with validation before running
